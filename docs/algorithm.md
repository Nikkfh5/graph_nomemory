# PageRank для графа больше RAM: алгоритм и гарантии

Это короткое описание решения для проверяющего. Здесь собраны выбранная
метрика, организация out-of-core обработки, поведение на гипер-узле степени
`50 000`, сложность в лучшем, типичном и худшем случаях, точность и
ограничения. Инструкции запуска находятся в [`README.md`](../README.md).

## Метрика

Реализован standard uniform PageRank для невзвешенного ориентированного графа:

\[
r_{k+1}(v)=\frac{1-\alpha}{V}+\alpha\left(
\sum_{u\to v}\frac{r_k(u)}{d^+(u)}+\frac{D_k}{V}
\right).
\]

`d⁺(u)` — исходящая степень, `D_k` — суммарный rank dangling-вершин без
исходящих рёбер. По умолчанию `alpha=0.85`; начальный вектор и teleportation
равномерны.

PageRank даёт каждой вершине глобальную оценку структурной важности. Вклад
ссылки зависит от rank источника и числа его исходящих рёбер. Метрика полезна
для web- и citation-графов, knowledge graphs и графов зависимостей.

PageRank выбран потому, что:

- является метрикой всего графа;
- вычисляется последовательными sparse matrix-vector проходами;
- допускает bounded disk-backed layout;
- имеет единственный fixed point и проверяемый residual.

## Почему граф не требуется держать в RAM

Используется **semi-external destination pull**. Рёбра, входящие степени и
каталог задач находятся на диске. В RAM остаётся состояние на вершину:

\[
8V\;(current)+8V\;(scratch)+4V\;(out\_degree)=20V\ \text{bytes}.
\]

Preflight дополнительно учитывает I/O-буферы, стеки и guard pages потоков,
scheduler window, служебные структуры и runtime reserve. Запуск разрешается
только при соблюдении memory budget.

На итерации destination layout читается последовательно. Разные задачи пишут
rank разных вершин. Конкурентные floating-point updates не нужны, а
фиксированный порядок редукций делает результат воспроизводимым.

Подход именно semi-external. Если `20V` со служебным резервом не помещается в
RAM, текущая версия отклоняет запуск. Fully-external 2D layout рассмотрен как
альтернатива, но не реализован как runtime fallback.

## Гипер-узел на 50 000 рёбер

Большая входящая adjacency делится на slices не более `edge_slice_size`
рёбер. При slice `8192` вершина с `50 000` входящих рёбер создаёт семь задач:

```text
6 × 8192 + 848 = 50 000
```

Workers считают bounded partial sums. Coordinator складывает их строго по
`slice_index`. Поэтому:

- работа для вершины остаётся `Theta(50 000)`;
- дополнительная RAM определяется scheduler window, не полной степенью;
- slices одной длинной adjacency могут обрабатываться параллельно;
- результат не зависит от порядка завершения workers.

Проверочный skew-граф имел `V=1 000 000`, `E=50 000 000`, out-hub степени
`50 000` и три in-hubs с физическим разбиением `7/3/3`. Анализ под hard limit
`128 MiB` завершился без OOM; последовательный и параллельный результаты
совпали. Сводная таблица наблюдений находится в разделе
[`README: Корректность`](../README.md#корректность).

## Сложность

Обозначения:

- `R` — корректные входные строки до удаления дубликатов;
- `E` — уникальные рёбра;
- `V` — вершины;
- `Q` — физические задачи, включая hub slices;
- `K` — candidate iterations;
- `C` — дополнительные true-residual passes, `0 <= C <= K`;
- `F` — fan-in внешнего merge;
- `N_v`, `N_e` — числа начальных runs вершин и рёбер.

### Подготовка

Потоковый parse требует `Theta(R)`. Binary-search remap требует
`O(R log V)`. Chunk sorting и external merge дают worst-case
`O(R log R)`. Безопасная общая верхняя оценка CPU:

\[
O(R\log R+R\log V).
\]

External record traffic:

\[
O\left(R(1+\log_F N_v+\log_F N_e)+E+V\right).
\]

В худшей оценке участвует `R`, а не `E`: дубликаты могут дать `R >> E`.
При `R≈E` и малом числе runs фактических merge-levels и I/O меньше.
Реализация всё равно сортирует bounded chunks, поэтому линейный best-case
для всей подготовки не заявляется.

### Итеративный PageRank

Одна candidate iteration и один residual pass требуют
`Theta(V+E+Q)` работы. Полный анализ:

\[
\Theta((K+C)(V+E+Q)).
\]

Один destination pass логически читает
`4E + 4V + size(tasks)` bytes, кроме одноразовой validation и `O(V)` output.

| Случай | Проходы | Стоимость анализа | Интерпретация |
| --- | --- | --- | --- |
| Лучший достижимый | `K=1`, `C=1` | `Theta(V+E+Q)` | Uniform start уже fixed point; один pass строит кандидата, второй проверяет residual |
| Типичный | Зависит от topology, `alpha`, spectral gap | `Theta((K+C)(V+E+Q))` | Универсальной average-case оценки без распределения графов нет; измерено `K=1,C=1` на regular scale и `K=94,C=7` на skew |
| Худший при defaults | `K=200`, возможно `C=200` | До `400` полных passes | Достигнут iteration cap; результат не публикуется как сошедшийся |

Resident RAM определяется `V`, числом потоков и bounded buffers, а не `E`
или максимальной степенью. Случаи выше меняют прежде всего число полных чтений
disk-backed graph.

## Точность

PageRank operator является L1 contraction с коэффициентом `alpha`. В точной
арифметике для кандидата `r`:

\[
\lVert r-r^*\rVert_1\le
\frac{\lVert F(r)-r\rVert_1}{1-\alpha}.
\]

Изменение между итерациями используется только как prefilter. Публикация
разрешается после отдельного прохода по всему графу и проверки:

\[
true\_residual+\epsilon_{verify}\le(1-\alpha)\eta,
\]

где по умолчанию `eta=1e-8`, `epsilon_verify=2^-30`. Некорректная масса,
`NaN`, `Inf`, отрицательный rank или invalid residual дают явный numerical
failure.

На малых fixtures максимальное отличие от независимого high-precision direct
solve составило `5.1025e-10 < 1e-8`. Численная гарантия ограничена
зафиксированным binary64-профилем и `V<=1 000 000`.

## Многопоточность

Используется persistent bounded pthread pool. Задачи исполняются waves без
неограниченной очереди. Ошибка выбирается детерминированно по минимальному
индексу задачи, затем executor останавливает и присоединяет workers.

На зарегистрированном 64-CPU стенде измерено:

- `17.73x` для parallel core;
- `1.15x` для полной analyze command.

Validation и запись CSV остаются serial/I/O bottlenecks. Линейный speedup,
полная загрузка CPU и переносимость конкретных timings не заявляются.

## Ограничения

- Нет автоматического fully-external fallback.
- Нет distributed/GPU режима.
- Timing зависит от host, filesystem, storage и cache state.
- SSD и cold-cache характеристики не заявляются.
- Numerical guarantee не расширяется за пределы проверенного профиля.
- Edge-only CSV не может задать вершину, отсутствующую во всех рёбрах.
