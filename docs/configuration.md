# Запуск и конфигурация `main`

## Публичный контракт

```text
main INPUT.csv OUTPUT.csv
```

- `INPUT.csv` — существующий обычный файл с ориентированными рёбрами; последний компонент пути не должен быть символической ссылкой.
- `OUTPUT.csv` — ещё не существующий путь результата `vertex,rank` внутри существующего обычного каталога. Результат публикуется атомарно и не заменяет существующий файл.
- `main --help` печатает краткую справку по синтаксису запуска и ссылку на этот документ.

Путь к файлу конфигурации фиксирован: `./config/main.conf` относительно текущего рабочего каталога процесса. В командной строке он не передаётся. Программа не читает переопределения из переменных окружения и не ищет системные или пользовательские файлы конфигурации.

Если `./config/main.conf` отсутствует (`ENOENT`), используются встроенные значения по умолчанию. Если файл существует, но недоступен для чтения, имеет неверный формат, является символической ссылкой в последнем компоненте пути или не является обычным файлом, это считается явной ошибкой: автоматический переход к значениям по умолчанию запрещён. В Docker рабочий каталог — `/data`, а образ содержит `/data/config/main.conf`.

Пути к входному и выходному файлам через конфигурацию не задаются.

## Файл конфигурации

Файл [`config/main.conf`](../config/main.conf) содержит все 35 изменяемых значений по умолчанию и автоматически применяется при запуске из корня репозитория. Чтобы изменить параметр, замените его значение после `=`:

```text
schema=tbank-run-config-v1
parallel.worker_count=1
pagerank.max_iterations=300
```

Основные правила:

- первая содержательная строка — `schema=tbank-run-config-v1`;
- остальные содержательные строки имеют вид `ключ=значение`; пробелы внутри них запрещены;
- пустые строки и строки-комментарии, начинающиеся с `#`, игнорируются;
- файл использует ASCII, заканчивается LF и не превышает `65536` байт;
- неизвестные и повторяющиеся ключи, неверные значения и переполнение считаются ошибкой.

Программа проверяет конфигурацию и доступные ресурсы до создания временных данных и выходного файла.

#### Как устроен эталонный набор параметров параллельного выполнения

Конфигурация версии 1 не определяет число CPU автоматически. `parallel.worker_count=64` зафиксирован как часть воспроизводимого набора параметров, проверенного на стенде Linux x86-64 с 64 доступными логическими процессорами. На машине с меньшим числом доступных CPU значение в хранящемся в репозитории файле `config/main.conf` следует явно уменьшить до результата `nproc`; масштабирование выше 64 рабочих потоков не исследовалось. Явное число рабочих потоков делает запуск повторяемым и позволяет до начала вычислений точно проверить потребление памяти, вместо того чтобы молча менять набор ресурсных параметров в зависимости от машины. Эти значения проверены как параметры с явными ресурсными ограничениями, но не подобраны автоматически и не являются универсально оптимальными.

Остальные параметры заданий и параллельного выполнения задают ограниченную структуру вычислений, а не коэффициенты алгоритма PageRank:

- `preprocess.max_task_edges=262144` и `preprocess.max_task_vertices=4096` ограничивают обычное задание дисковой обработки одновременно по числу рёбер и вершин, чтобы одна задача не захватывала неограниченный диапазон;
- `parallel.worker_stack_bytes=1048576` и `parallel.worker_guard_bytes=4096` задают явно учитываемые область памяти под стек и защитную страницу каждого рабочего потока; значения проверяются относительно `PTHREAD_STACK_MIN` и фактического размера страницы системы;
- `parallel.worker_count_batch_records=8192` и `parallel.worker_source_batch_records=8192` ограничивают чтение `incoming_counts` и `incoming_sources` для каждого рабочего потока; при четырёхбайтовой записи это два буфера по `32768` байт на поток;
- `parallel.scheduler_window_records=64` ограничивает число записей заданий и частичных сумм, одновременно находящихся у координатора; фактический параллелизм не превышает минимум из `parallel.worker_count`, окна планировщика и числа готовых заданий, поэтому увеличение только `parallel.worker_count` выше `64` не расширяет пакет заданий по умолчанию;
- `preprocess.edge_slice_size=8192` ограничивает одно задание для большого списка входящих рёбер: степень `50000` превращается в семь фрагментов `6 × 8192 + 848`, которые можно обрабатывать параллельно без буфера на всю вершину высокой степени.

Для `T=parallel.worker_count` и `W=parallel.scheduler_window_records` пиковое потребление памяти параллельным этапом рассчитывается по явной формуле с верхней границей:

```text
M_parallel =
  T * (worker_stack + worker_guard + 4 * count_batch + 4 * source_batch)
  + executor_control(T)
  + W * (32 байта на запись задания + 8 байт на частичную сумму)

required =
  20 * V
  + max(пик однопоточного ввода-вывода, M_parallel)
  + фактический RLIMIT_STACK
  + резерв среды выполнения 32 MiB
  <= 128 MiB
```

Здесь `V` — число вершин, а `executor_control(T)` — отдельно рассчитанные служебные структуры постоянного набора рабочих потоков. Все умножения и сложения выполняются с проверкой переполнения.

План памяти с проверкой допустимости ресурсных параметров рассчитывается до выделения `O(V)` памяти и чтения рёбер. Сам набор рабочих потоков создаётся лениво после выделения памяти под состояние вершин; до первого пакета параллельных заданий каждый рабочий поток в Linux проверяет фактические области памяти под стек и защитную страницу. Несовпадение с планом останавливает запуск до выполнения параллельных заданий обработки рёбер.

Размер стека основного потока не задаётся пользователем: программа читает фактический конечный мягкий предел `RLIMIT_STACK` и учитывает его при предварительной проверке памяти. Неограниченный стек отклоняется. Для последовательного режима нужно одновременно задать `parallel.worker_count=0` и остальные пять ресурсных параметров `parallel.*` равными `0`; неполная комбинация отклоняется.

### PageRank

| Ключ | Значение по умолчанию |
| --- | ---: |
| `pagerank.alpha` | `0.85` |
| `pagerank.eta` | `1e-8` |
| `pagerank.max_iterations` | `200` |

`alpha` должен быть конечным числом из `(0,1)`, `eta` — конечным положительным числом, `max_iterations` — положительным целым числом. Параметры буферов, стеков, числа одновременно сливаемых серий, заданий и резервов проходят расчёт ресурсов с проверкой допустимости; конфликтующие или непредставимые сочетания отклоняются до начала обработки.

## Ограничение 128 MiB

`memory_budget_bytes` пока не является настраиваемым параметром. Предварительная обработка и PageRank строго соблюдают предел `134217728` байт (`128 MiB`), а численные эксперименты покрывают проверенную область `V<=1,000,000`. Ключ конфигурации, добавленный только для вида, создал бы ложную настраиваемость: изменение предела памяти требует параметризации обоих расчётов ресурсов и новых проверок. Конфигурация версии 1 оставляет эту работу отдельной будущей задачей.

## Временные данные и публикация

После предварительных проверок `main` создаёт собственный каталог `.tbank-run-*` рядом с выходным файлом, выполняет предварительную обработку и PageRank, затем удаляет только этот каталог. Обычная ошибка очищает промежуточные файлы и не публикует результат. Аварийное завершение может оставить отличимый каталог `.tbank-run-*`; он не считается завершённым результатом.

Если после `rename` завершается ошибкой заключительный `fsync` родительского каталога, выходной файл может уже быть виден, но его сохранность после сбоя не подтверждена. Процесс возвращает отдельный ненулевой код и не пытается удалить уже опубликованный файл.

## Коды завершения и ошибки

При успешном запуске программа ничего не печатает: результат находится в `OUTPUT.csv`. Диагностическое сообщение записывается в `stderr` одной понятной человеку строкой.

| Код | Класс |
| ---: | --- |
| `0` | успешное завершение |
| `2` | ошибка вызова или конфигурации |
| `3` | ошибка входного файла, данных или формата |
| `4` | ошибка предварительной проверки или выделения ресурсов |
| `5` | отсутствие сходимости |
| `6` | ошибка численной проверки |
| `7` | ошибка публикации или отказ от замены существующего файла |
| `8` | сохранность опубликованного файла после сбоя не подтверждена |
| `9` | системная ошибка операционной системы или ввода-вывода |
| `10` | нарушение внутреннего инварианта или внутренняя ошибка |

Формат входа и выхода кратко описан в [`README.md`](../README.md), алгоритм и ресурсные ограничения — в [`algorithm.md`](algorithm.md), область применимости экспериментов — в [`research.md`](../research.md).
