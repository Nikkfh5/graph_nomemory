# Журнал исследования

## Исходная задача

Требовалось вычислять аналитическую меру для большого ориентированного графа
на одном узле. Полное рабочее представление графа могло превышать доступную
оперативную память. Решение также должно было учитывать вершины с очень
большой степенью, использовать несколько потоков и давать воспроизводимый
результат.

Для реализации выбран PageRank. Он даёт одно число каждой вершине, учитывает
значимость входящих связей и сводится к повторным проходам по рёбрам.

## Рассмотренные способы

### Полувнешний обход по входящим рёбрам

Рёбра хранятся на диске, а значения ранга и степени исхода вершин находятся в
памяти. Каждая вершина получает результат независимо от других вершин текущей
итерации.

Преимущества:

- простой контроль памяти;
- последовательное чтение рёбер;
- естественное разделение работы между потоками;
- не требуется записывать промежуточное обновление для каждого ребра.

Этот способ выбран для реализации.

### Полностью внешний блочный обход

В памяти находится только часть вершин, а граф многократно читается блоками.
Такой способ нужен, если даже состояние размером `20V` байт не помещается в
память.

Он не выбран из-за большего числа проходов по диску, сложной публикации
промежуточных состояний и более трудного восстановления после сбоя.

### Обход по исходящим рёбрам

Каждая исходная вершина создаёт поток обновлений для соседей. Затем обновления
нужно сортировать или собирать по вершинам назначения.

Способ отклонён из-за дополнительной записи объёма порядка числа рёбер и
неудобного воспроизводимого сложения в нескольких потоках.

## Основные найденные проблемы

### Граф нельзя считать обычным массивом рёбер

Для набора из миллиона вершин и сорока миллионов рёбер одно только привычное
представление графа превышало предел `128 MiB`.

Решение: рёбра и каталог заданий оставлены на диске. В памяти находятся два
массива `double[V]` и массив степеней `uint32[V]`, всего `20V` байт до
ограниченных служебных расходов.

### Нельзя держать таблицу смещений и каталог заданий в памяти

Обычный разреженный формат добавлял ещё один большой массив смещений.
Физических заданий также могло стать слишком много.

Решение: число входящих рёбер хранится на диске для каждой вершины, а каталог
заданий читается последовательно ограниченными порциями.

### Идентификаторы вершин могут быть разреженными

Исходные значения `int32` нельзя использовать как индексы массивов.

Решение: подготовка строит отсортированный словарь исходных идентификаторов и
перенумеровывает их в плотный диапазон. Результат затем записывается с
исходными значениями.

### Входной файл нельзя сортировать в памяти

Число строк заранее не ограничено размером оперативной памяти.

Решение: вход разбивается на ограниченные серии. Серии сортируются отдельно,
а затем объединяются с ограниченным числом одновременно открытых файлов.
Повторяющиеся рёбра удаляются при окончательном объединении.

### Гипер-узел нарушает обычное разбиение

Одна вершина может иметь существенно больше входящих рёбер, чем средняя.
Передача всей её окрестности одному потоку создаёт большой буфер и плохой
баланс.

Решение: список входящих рёбер делится на срезы. Частичные суммы вычисляются
независимо и складываются координатором в постоянном порядке.

### Параллельное сложение меняло бы результат

Сложение чисел с плавающей точкой зависит от порядка операций. Общая
атомарная сумма сделала бы результат зависимым от порядка выполнения потоков.

Решение: вершины имеют единственного владельца записи, а частичные суммы
гипер-узлов сводятся в заранее определённой последовательности. Параметры
компилятора, меняющие правила вычислений, запрещены.

### Малого изменения между итерациями недостаточно

Небольшая разность соседних векторов ещё не гарантирует требуемую точность
уравнения PageRank.

Решение: изменение используется только как предварительное условие. Перед
публикацией отдельно вычисляется настоящая невязка. Также проверяется сумма
вероятностей и выполняется нормировка.

### Ограничение памяти нельзя доказать только подсчётом массивов

На расход влияют стеки потоков, буферы чтения, служебные данные библиотек и
страничный кэш файловой системы.

Решение: перед чтением крупных файлов вычисляется полный план памяти.
Отдельно учитываются основной стек, стеки потоков, защитные страницы, буферы
и обязательный запас. Неподдерживаемая конфигурация отклоняется заранее.

### Отображение файла в память не решает задачу само по себе

Страницы отображённого файла всё равно учитываются ядром и могут вытеснить
другие данные или привести к превышению ограничения.

Решение: используются ограниченные позиционные чтения. Проверка памяти
опирается на общий расход процесса, а не на размер его обычной кучи.

### Успешная запись ещё не означает надёжную публикацию

Сбой между записью файла, переименованием и синхронизацией каталога мог
оставить неоднозначное состояние.

Решение: данные сначала пишутся во временное имя, синхронизируются, затем
публикуются без замены существующего результата. Ошибка после переименования,
но до подтверждения каталога, имеет отдельное состояние неопределённой
долговечности.

### Внешнее наблюдение не доказало одновременную работу потоков

Периодический просмотр `/proc` пропускал короткие участки вычислений и не мог
надёжно отличить последовательное выполнение от параллельного.

Решение: исполнитель измеряет время процессора каждого рабочего потока
внутри границ вычислительной волны. Это показало перекрытие работы потоков,
но не используется как заявление о полной загрузке всех ядер.

### Контейнерная сборка зависела от сети стенда

Первая локальная сборка не смогла разрешить адреса хранилищ пакетов Ubuntu
внутри сети Docker.

Решение: на этом стенде получение пакетов выполнено с `--network host`.
Рабочая программа не требует сети. Сборка в GitHub завершилась обычным
способом.

## Наблюдаемые результаты

| Проверка | Результат |
| --- | --- |
| Малые графы и независимое решение системы | наибольшая ошибка L1 `5.1025e-10` при пороге `1e-8` |
| Миллион вершин и сорок миллионов рёбер | дисковый граф больше `128 MiB`, подготовка и анализ завершены без нехватки памяти |
| Миллион вершин и пятьдесят миллионов рёбер | вершины степени `50 000` обработаны через срезы |
| Один и 64 потока | результаты совпали побитово |
| Основная параллельная часть | ускорение медианы `17.73x` |
| Полная команда анализа | ускорение `1.15x` из-за проверки графа и записи CSV |
| Ubuntu 22.04 в Docker | три исполняемых файла успешно собраны |

Числа относятся к конкретному узлу Linux x86-64 и указанным наборам данных.
Они не обещают ту же скорость на другом накопителе или процессоре.

## Границы решения

- Основное состояние `20V` должно помещаться в память вместе со служебными
  расходами.
- Полностью внешний блочный способ не включается автоматически.
- Входной CSV не задаёт изолированные вершины.
- Дисковое представление не сжато.
- Результаты для SSD и холодного кэша не заявляются.
- Насыщение выше 64 потоков не исследовалось.
- Численная область стандартных параметров проверена до миллиона вершин.
- Решение предназначено для одного узла Linux x86-64.

## Итог

Выбранный способ переносит зависимость от числа рёбер из оперативной памяти на
диск, сохраняет ограниченное состояние на вершину и допускает
детерминированное многопоточное выполнение. Главный предел остаётся линейным
по числу вершин: если `20V` с запасами не помещается, потребуется отдельная
полностью внешняя архитектура.
