# Graph NoMemory

Вычисление PageRank для большого ориентированного графа на одном узле при
жёстком ограничении оперативной памяти.

Проект не загружает граф целиком в память. Рёбра и каталог вычислительных
заданий остаются на диске, а в оперативной памяти находятся два массива
значений PageRank, степени исхода вершин и заранее ограниченные буферы.
Основной алгоритм, внешняя подготовка графа, проверка формата, разбиение
гипер-узлов и многопоточный исполнитель реализованы на C++20 без готового
графового движка.

Короткий документ для проверяющего с явным разбором гипер-узлов, точности,
лучшего, типичного и худшего случаев:
[`docs/algorithm.md`](docs/algorithm.md).

## Что находится в репозитории

```text
app/                         команды подготовки, анализа и проверки заданий
include/tbank/               открытые заголовочные файлы компонентов
src/                         реализация хранения, подготовки и PageRank
tools/generators/generate.py детерминированный генератор проверочных графов
CMakeLists.txt               сборка трёх исполняемых файлов
Dockerfile                   сборка компактного рабочего образа
```

## Задача и выбранная метрика

Для `n = |V| > 0`, коэффициента затухания `0 < alpha < 1` и равномерного
распределения переходов вычисляется стандартный PageRank:

```text
r(k+1, v) = (1 - alpha) / n
          + alpha * (
              sum(r(k, u) / out_degree(u), u -> v)
              + dangling_mass(k) / n
            )
```

Начальное значение каждой вершины равно `1/n`. По умолчанию используются
`alpha=0.85`, требуемая точность `eta=1e-8` и не более `200` итераций.
Масса висячих вершин, у которых нет исходящих рёбер, распределяется
равномерно.

PageRank даёт каждой вершине меру глобальной структурной значимости:
ссылка от уже значимой вершины имеет больший вес. Метрика применяется к
веб-графам, графам цитирования, знаний и зависимостей. Она выбрана потому,
что естественно вычисляется последовательными проходами по входящим рёбрам,
даёт одно значение каждой вершине и допускает независимую проверку на малых
графах.

## Рассмотренные способы обработки

| Способ | Память и ввод-вывод | Итог |
| --- | --- | --- |
| Полувнешний обход по входящим рёбрам | `20V` байт основного состояния; рёбра читаются с диска | Реализован: простая модель памяти, независимая запись результатов по вершинам |
| Полностью внешний блочный обход | В памяти только блок вершин; больше повторных чтений и промежуточных состояний | Оставлен как возможное развитие, если `20V` не помещаются |
| Обход по исходящим рёбрам с потоком обновлений | Требует записывать, сортировать и сводить обновления для рёбер | Отклонён из-за лишнего дискового трафика и более сложного восстановления после ошибок |

Текущая реализация является полувнешней. Она заранее отклоняет граф, если
даже ограниченное состояние `20V` вместе с буферами и стеками не помещается
в заданный предел памяти.

## Устройство решения

```text
edges.csv
  -> строгий потоковый разбор
  -> внешняя сортировка и объединение идентификаторов
  -> перенумерация, сортировка и удаление повторов рёбер
  -> неизменяемый граф на диске и каталог заданий
  -> многопоточный PageRank по входящим рёбрам
  -> проверка невязки
  -> надёжная публикация pagerank.csv
```

Подготовка выполняется командой `tbank-prepare`. Она строит упорядоченный
словарь исходных идентификаторов, переводит рёбра в компактные номера,
глобально удаляет повторы и публикует каталог графа только после проверки
размеров, счётчиков и контрольных сумм.

Команда `tbank-analyze` проверяет каталог графа, рассчитывает PageRank и
записывает итоговый CSV. Два массива `double[V]` переиспользуются между
итерациями; отдельный массив вкладов и таблица смещений в памяти не нужны.
Базовое состояние анализа занимает:

```text
current rank  8V
next rank     8V
out degree    4V
----------------
total        20V bytes
```

Полная предварительная проверка дополнительно учитывает буферы чтения,
стеки потоков, окно планировщика, служебные структуры и заданный запас.

### Гипер-узлы

Большой список входящих рёбер одной вершины делится на срезы фиксированного
размера. Потоки независимо считают частичные суммы, после чего координатор
складывает их в неизменном порядке. Поэтому максимальная степень вершины не
определяет размер одного буфера и не нарушает воспроизводимость результата.

### Многопоточность

Используется постоянный набор потоков POSIX и ограниченные волны заданий без
неограниченной общей очереди. Обычная вершина имеет ровно один поток записи,
а частичные суммы гипер-узлов сводятся в фиксированном порядке. Атомарное
сложение чисел с плавающей точкой и параметры `-ffast-math`/`-Ofast`
запрещены.

## Требования

Поддерживаемая среда:

- Linux x86-64, проверочная основа Ubuntu 22.04;
- компилятор C++20, например GCC 11;
- CMake не ниже 3.22 и Make;
- заголовочные файлы OpenSSL Crypto не ниже 3.0;
- Python не ниже 3.10 только для синтетического генератора;
- Docker по желанию.

Пакеты для Ubuntu 22.04:

```bash
sudo apt-get update
sudo apt-get install --yes --no-install-recommends \
  cmake g++ libssl-dev make python3
```

## Сборка

```bash
git clone https://github.com/Nikkfh5/graph_nomemory.git
cd graph_nomemory

cmake -S . -B build \
  -G "Unix Makefiles" \
  -DCMAKE_BUILD_TYPE=Release \
  -DCMAKE_CXX_COMPILER=/usr/bin/g++
cmake --build build --parallel
```

Будут созданы:

- `build/tbank-prepare` — преобразование CSV в дисковый граф;
- `build/tbank-analyze` — вычисление PageRank и запись результата;
- `build/tbank-inspect-tasks` — проверка физического разбиения заданий.

Каждая команда поддерживает `--help`.

## Генератор синтетических графов

Чтобы получить готовый входной CSV без ручного составления рёбер, запустите
генератор из корня checkout:

```bash
python3 tools/generators/generate.py \
  --profile reduced-skew \
  --output ./dataset
```

Генератор создаёт новый каталог `dataset` с двумя файлами:

- `edges.csv` — ориентированный граф в готовом для `tbank-prepare` формате;
- `manifest.json` — профиль, число вершин и рёбер, SHA-256 и Git revision
  генератора.

Git revision определяется автоматически, передавать SHA в командной строке
не нужно. Сейчас поддерживаются фиксированные воспроизводимые профили;
произвольные количество вершин, количество рёбер и плотность не задаются.
Полный список профилей, проверка результата и локальный режим описаны в
[`tools/generators/README.md`](tools/generators/README.md).

## Быстрый сквозной пример

Пример создаёт небольшой детерминированный граф из `101` вершины и `404`
уникальных рёбер, готовит дисковое представление, проверяет задания и
запускает два вычислительных потока.

```bash
work_dir="$(mktemp -d -p "${TMPDIR:-/tmp}" graph-nomemory.XXXXXX)"

python3 tools/generators/generate.py \
  --profile reduced-skew \
  --output "$work_dir/dataset" \
  --chunk-bytes 127

input_sha256="$(sha256sum "$work_dir/dataset/edges.csv" | cut -d ' ' -f 1)"

build/tbank-prepare \
  --input-csv "$work_dir/dataset/edges.csv" \
  --input-sha256 "$input_sha256" \
  --workspace "$work_dir/workspace" \
  --graph-dir "$work_dir/graph" \
  --non-bulk-reserve-bytes 33554432 \
  --input-chunk-bytes 128 \
  --phase-fd-budget 64 \
  --disk-reserve-bytes 0 \
  --edge-slice-size 4 \
  --max-task-edges 4 \
  --max-task-vertices 8

build/tbank-inspect-tasks \
  --graph-dir "$work_dir/graph" \
  --validation-chunk-bytes 53

(
  ulimit -S -s 1024
  build/tbank-analyze \
    --graph-dir "$work_dir/graph" \
    --output-csv "$work_dir/pagerank.csv" \
    --worker-count 2 \
    --record-batch-records 16 \
    --validation-chunk-bytes 4096 \
    --main-stack-bytes 1048576 \
    --runtime-reserve-bytes 16777216 \
    --worker-stack-bytes 1048576 \
    --worker-guard-bytes 4096 \
    --worker-count-batch-records 16 \
    --worker-source-batch-records 16 \
    --scheduler-window-records 8
)

head "$work_dir/pagerank.csv"
printf 'Результат: %s\n' "$work_dir"
```

Целевые каталоги и файл результата не должны существовать заранее. Успешные
команды подготовки и анализа печатают в стандартный вывод одну строку JSON с
параметрами и итогом операции.

## Форматы данных

### Входной CSV

```csv
from,to
10,20
20,30
30,10
```

Требования:

- кодировка UTF-8 без BOM;
- точный заголовок `from,to`;
- одна строка означает ориентированное ребро `from -> to`;
- идентификаторы входят в диапазон `int32`;
- каноническая запись числа: `0` или `-?[1-9][0-9]*`;
- `-0`, знак `+`, ведущие нули, кавычки, пустые и лишние поля запрещены;
- повторяющиеся рёбра удаляются, петли сохраняются;
- множество вершин равно объединению концов рёбер, поэтому изолированную
  вершину таким форматом задать нельзя.

Ожидаемая SHA-256 передаётся явно. Это защищает длительную подготовку от
незаметной подмены или изменения входного файла.

### Каталог графа

Промежуточное представление имеет версионные двоичные заголовки, размеры,
контрольные суммы CRC-64/ECMA-182 и канонический `metadata.json`. Полезная
часть занимает приблизительно:

```text
4E + 12V + размер каталога заданий + служебные данные
```

Каталог считается готовым только после полной проверки и публикации
`metadata.json`. Незавершённый временный каталог не принимается анализатором.

### Результат

```csv
vertex,rank
10,0.33333333333333331
20,0.33333333333333331
30,0.33333333333333331
```

Исходные идентификаторы записываются по возрастанию. Значения конечны,
выводятся в локали C с точностью, достаточной для обратного чтения `double`.
Файл сначала полностью записывается и синхронизируется во временное имя,
затем публикуется атомарно без перезаписи существующего результата.

## Корректность

Каждая итерация проверяет сумму вероятностей, нормирует следующий вектор и
считает изменение по норме L1. Достижение порога изменения является только
предварительным условием: перед публикацией отдельно вычисляется настоящая
невязка уравнения PageRank.

Для профиля по умолчанию публикация требует:

```text
true_residual + 2^-30 <= (1 - alpha) * eta
```

Проверенная численная область профиля по умолчанию ограничена
`V <= 1 000 000`. Для больших графов программа должна получить отдельно
обоснованные численные параметры, а не молча расширять доказанную область.

Перед подготовкой публичного репозитория исходная реализация прошла следующие
проверки:

| Проверка | Наблюдаемый результат |
| --- | --- |
| Малые графы и независимое решение системы | максимальная ошибка L1 `5.1025e-10`, меньше `1e-8` |
| Ограничение памяти, `V=1 000 000`, `E=40 000 000` | дисковый граф `172 009 165` байт больше лимита `128 MiB`; подготовка и анализ завершились без OOM |
| Граф с гипер-узлами, `V=1 000 000`, `E=50 000 000` | степени гипер-узлов `50 000`; последовательный и параллельный результаты совпали |
| Один и 64 потока | восемь сравнений дали побитовое совпадение рангов |
| Параллельная основная часть | ускорение медианы `17.73x`; полная команда анализа `1.15x` из-за проверки и записи CSV |
| Чистая Ubuntu 22.04 | сборка контейнера и все закрытые проверки завершились успешно |

Это наблюдения конкретного Linux x86-64 стенда, а не обещание такой же
скорости на другом оборудовании.

## Оценка сложности

Развёрнутое объяснение обозначений, I/O и лучшего/типичного/худшего случаев:
[`docs/algorithm.md`](docs/algorithm.md#сложность).

Пусть `R` — число входных строк до удаления повторов, `E` — число уникальных
рёбер, `V` — число вершин, `Q` — число физических заданий, `K` — число
итераций, `C` — число дополнительных проверок невязки.

- подготовка: `O(R log R + R log V)` вычислений при внешней сортировке;
- одна итерация или проверка невязки: `Theta(V + E + Q)`;
- анализ: `Theta((K + C)(V + E + Q))`;
- основное состояние в памяти: `20V` байт плюс заранее ограниченные расходы;
- постоянный граф на диске: `4E + 12V + O(Q)` байт;
- число уровней внешнего слияния зависит от числа начальных серий и
  допустимого количества одновременно открытых файлов.

## Docker

Локальная сборка рабочего образа:

```bash
docker build --tag graph-nomemory:local .
docker run --rm graph-nomemory:local
```

В образ попадают только три исполняемых файла и необходимые рабочие
библиотеки. Компилятор, исходный код, Python и промежуточная сборка остаются
в предыдущем слое и отсутствуют в итоговом образе.

После отправки изменений в `main` GitHub Actions собирает тот же `Dockerfile`
и публикует образ:

```bash
docker pull ghcr.io/nikkfh5/graph_nomemory:main
```

## Коды завершения

| Код | Значение |
| ---: | --- |
| `0` | успешное выполнение |
| `2` | ошибка аргументов или их записи |
| `3` | неверный или изменившийся входной файл либо каталог графа |
| `4` | недостаточно памяти, места, дескрипторов или неподходящие стеки |
| `5` | предел итераций достигнут без подтверждённой сходимости |
| `6` | нарушение численных условий |
| `7` | результат не удалось надёжно опубликовать |
| `8` | имя результата появилось, но долговечность записи не подтверждена |
| `9` | ошибка операционной системы |
| `10` | нарушен внутренний инвариант |

Проверять нужно и код завершения, и поле состояния в строке JSON.

## Ограничения

- Решение работает на одном узле Linux x86-64 и не использует кластер или
  графический ускоритель.
- Если `20V` вместе с ограниченными расходами не помещается в память,
  полностью внешний запасной способ автоматически не включается.
- Входной CSV не умеет задавать изолированные вершины.
- Дисковый граф не сжат; обход может иметь слабую локальность чтения значений
  исходных вершин.
- Время зависит от процессора, файловой системы, накопителя и формы графа.
  Результаты для SSD и холодного кэша не заявляются.
- Насыщение выше 64 потоков не исследовалось.
- Надёжность публикации зависит от поведения `fsync` и
  `renameat2(RENAME_NOREPLACE)` в используемой файловой системе.
